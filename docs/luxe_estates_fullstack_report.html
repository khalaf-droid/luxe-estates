Part 1 — Critical Compatibility Bugs (Break Everything Silently)
FATAL
API base URL — wrong port AND missing /v1/ prefix
environment.ts → http://localhost:5000/api · Backend runs on port 3000 with prefix /api/v1. Every single HTTP call in PropertiesService, AuctionsComponent, and AgentsComponent hits the wrong address and silently falls back to mock data. Meanwhile AuthService hardcodes localhost:3000/api/v1/auth correctly but ignores the environment file entirely — two different sources of truth.
FATAL
Property model — status field semantics are completely wrong
Frontend Property.status uses 'for-sale' | 'for-rent' and filters on it. Backend has two separate fields: listingType: 'sale' | 'rent' (the sale/rent distinction) and status: 'available' | 'reserved' | 'sold' (availability state). The filter tabs "For Sale" and "For Rent" will never match any real backend data. Frontend model, filter service, property cards, and mock data all need to be updated.
FATAL
Role names — frontend uses entirely different role vocabulary
NavComponent checks role === 'Buyer', role === 'Seller', role === 'Agent' (Title Case). Backend enum is buyer | owner | agent | admin (lowercase). 'Seller' does not exist in the backend at all. BecomeAgentComponent writes role = 'Seller' to localStorage. The correct upgrade role is 'owner' or 'agent'. Every role-based check in the nav and routing logic silently fails.
FATAL
Auction response shape mismatch — always falls to mock
AuctionsComponent calls this.http.get<Auction[]>('.../auctions') expecting a raw array. Backend returns { success: true, data: [...], pagination: {...} }. The response unwrapping is missing — result is always treated as an error → always falls back to mock data. Also: AuctionsComponent sends bids to /bids but backend expects /api/v1/bids (after the URL fix), and the Auction interface is missing startingPrice, bidIncrement, isApproved, status fields from the backend model.
FATAL
Favorites — wrong HTTP method signature for removal
Frontend sends POST /favorites/:propertyId and expects a boolean favorited toggle in one call. Backend uses separate endpoints: POST /favorites/:propertyId (add) and DELETE /favorites/:propertyId (remove). Frontend must check current state and choose method accordingly. Response shape is also different: backend returns { success, message } without a favorited field — so the toggle boolean will always be undefined.
FATAL
Viewing & Inquiry endpoints — wrong paths and field names
PropertiesService sends viewings to /viewings — backend path is /viewing-requests. Body sends { propertyId, date } — backend expects { property, preferredDate, preferredTime, message }. For inquiries, frontend sends { propertyId, message } — backend expects { property, content } (both field names differ). Both calls silently fail and return of(false).
FATAL
Agents endpoint — does not exist in the backend
AgentsComponent calls GET /api/v1/agents. This route does not exist in the backend (18 route files documented — no agents route). The component always falls back to hardcoded mock data. An agents endpoint must be added to the backend, or the frontend must query GET /users?role=agent,owner.
HIGH
No token refresh interceptor — sessions expire silently
Access tokens expire after 15 minutes. When a 401 is received, no interceptor calls POST /auth/refreshToken. The backend issues a refresh token in the httpOnly cookie and also in the JSON body. Frontend never uses either. Users are silently stuck with stale auth until they manually log out and back in.
HIGH
Property type enum — frontend is a subset of backend
Frontend type union: 'apartment' | 'villa' | 'penthouse' | 'estate'. Backend enum: apartment | villa | house | studio | office | shop | land | commercial. Penthouse and estate don't exist on the backend. Properties of types house/studio/office/shop/land/commercial will not render correctly in property cards (type badge will be blank or throw).

Part 4 — Backend Bugs (From Architecture Report)
C-01
Real credentials committed to .env in git
Gmail app password, MongoDB Atlas URI with credentials, JWT_SECRET, ENCRYPTION_KEY all committed. Must rotate all secrets immediately, add .env to .gitignore, purge from git history (BFG or git filter-repo), use GitHub Secrets in CI.
C-02
changeUserRole — .lean() + .save() runtime crash
dashboard.controller.js calls .lean() (returns plain JS object) then .save() — will throw "save is not a function". Fix: remove .lean() or use findByIdAndUpdate().
C-03
adminStats revenue — wrong status filter, always returns $0
Revenue aggregation matches status: "paid" — but Payment schema enum values are pending | completed | failed | refunded | expired. "paid" is not valid. Revenue will always be $0. Fix: change to status: "completed".
H-01
Rate limiter is in-memory only — fails in multi-instance deployments
Each Railway/Docker instance has its own counter. An attacker gets 20×N auth attempts across N instances. Fix: install rate-limit-redis and connect to the existing Redis client.
H-02
Refresh token exposed in JSON response body
Login returns the refresh token both in the httpOnly cookie AND in the JSON body. Frontend must not store it in localStorage. Web clients should use only the cookie. Fix: remove refresh token from JSON body for web responses.
H-03
Dashboard recentPayments — wrong populate field names
Populates user_id and booking_id but Payment model defines user and booking (no underscore). These fields will be null in the response. Fix: update to populate('user') and populate('booking').
